## 3. 목표 아키텍처 — 도메인 팩

### 3.1 개념

```
┌─────────────────────────────────────────────────────────┐
│ CORE (모든 프로젝트에 항상 로드)                          │
│  · 범용 에이전트 19 + 중립화된 하이브리드                  │
│  · 범용 skill (code-review, security-audit, sprint-plan…) │
│  · 범용 rule / 훅 / 템플릿                                 │
└─────────────────────────────────────────────────────────┘
        │                              │
        ▼                              ▼
┌──────────────────┐          ┌──────────────────────────┐
│ GAME-PACK        │          │ PRODUCT-PACK             │
│ (게임 감지 시만)  │          │ (웹/앱/서비스 감지 시만)  │
│  level-designer  │          │  product-manager         │
│  gameplay-prog.  │          │  frontend-engineer       │
│  world-builder   │          │  backend-engineer        │
│  live-ops-design │          │  mobile-engineer         │
│  game-designer   │          │  data-engineer           │
│  art-bible,      │          │  growth-engineer         │
│  balance-check…  │          │  technical-writer        │
│                  │          │  PRD, user-research,     │
│                  │          │  AB-test, scaffolds…     │
└──────────────────┘          └──────────────────────────┘
```

### 3.2 물리적 파일 레이아웃 (제안)

플러그인 디렉터리는 평면 구조(`agents/`, `skills/`)를 유지하되 — **메타데이터로 팩을 표시**(디렉터리 이동은 마켓플레이스 호환성 리스크). 각 에이전트/skill frontmatter에 `pack:` 필드 추가:

```yaml
---
name: level-designer
pack: game          # core | game | product
domain: [game]      # 활성 도메인 (복수 가능)
---
```

오케스트레이터와 `project-type-detect` 훅이 이 필드를 읽어 무관한 팩을 라우팅에서 제외. **파일은 안 옮기고 태그만** — 차기 세션 구현 부담 최소화, 점진적 마이그레이션 가능.

> 대안(2안): `agents/core/`, `agents/game/`, `agents/product/` 하위 디렉터리. 더 명확하지만 Claude Code 플러그인이 중첩 agents 디렉터리를 스캔하는지 검증 필요 — **구현 전 1차 확인 항목**.

## 4. project-type-detect — 자동 감지 (성능 핵심)

`hooks/detect-project-type.sh` 신규. SessionStart에서 1회 실행, 결과를 컨텍스트에 1줄 주입.

| 신호 | 판정 |
|---|---|
| `Assets/` + `ProjectSettings/` | `game` (Unity) |
| `*.godot` / `project.godot` | `game` (Godot) |
| `pubspec.yaml` | `mobile` (Flutter) |
| `app.json`+`expo` in package.json | `mobile` (React Native/Expo) |
| `next.config.*` / `package.json`+react | `web` |
| `requirements.txt`/`pyproject.toml`+fastapi/flask | `service` (백엔드) |
| AI 신호: `@anthropic-ai`, `google-generativeai`, `openai` 의존성 | `+ai` 플래그 |

출력 예: `PROJECT_TYPE=web+ai` → 오케스트레이터가 product-pack + ai 워크플로우만 로드, game-pack 무시.

**성능 효과:** 무관 에이전트/문서가 컨텍스트에 안 올라옴 → 토큰 절감 + 오케스트레이터 라우팅 정확도 상승. 기존 Unity opt-in 훅과 동일한 "감지→advisory" 패턴 재사용.

## 6. 코어 중립화 — 하이브리드 리네임 (앨리어스 유지)

게임 프로젝트에서 기존 이름이 계속 동작하도록 **앨리어스(구 이름 → 신 이름) 매핑** 유지.

| 현재(게임) | 코어 중립명 | 비고 |
|---|---|---|
| game-designer | `product-designer` | 게임 메커닉 부분은 game-pack의 `mechanics-designer`로 분리 |
| art-director | `design-lead` | 비주얼 아이덴티티 = 웹 디자인 시스템 |
| narrative-director | `content-strategist` | 스토리 = 콘텐츠 전략 |
| community-manager | `marketing-lead` | 패치노트 = 릴리스 커뮤니케이션 |
| writer | `content-writer` | UX 라이팅 포함 |

systems-designer / economy-designer / audio-director / sound-designer / technical-artist는 game-pack에 유지(앱/웹 수요 낮음). 필요 시 후속.

```yaml

  web-ai-saas:        # SpecForge / MoodReedAI 패턴
    framework: Next.js 16 (App Router)
    styling: Tailwind v4
    backend: Firebase (Firestore + Auth)

    payments: Stripe
  mobile-ai:          # ReviewSupporter / Invest_Log 패턴
    framework: React Native (Expo)
    backend: Firebase + Cloud Run proxy
    ai: Gemini
  game:               # StarDiver 패턴
    engine: Unity 6
    lang: C#
```
